Mastering C# 9 Logical Patterns: Using and, or, and not

2026-05-03

Introduction to C# 9 Logical Patterns

The evolution of the C# language has consistently focused on improving developer productivity and code readability. With the release of C# 9, Microsoft introduced a powerful feature set known as logical patterns. These patterns allow developers to combine existing patterns using the and, or, and not keywords. This shift fundamentally changes how we handle conditional logic, making complex validation and data filtering significantly more expressive.

Before C# 9, pattern matching was often limited to simple type checks or constant comparisons. If you needed to verify that a number fell within a specific range or that an object satisfied multiple criteria, you were often forced to write nested if-statements or complex boolean expressions. Logical patterns simplify these structures, enabling you to express your intent directly within the syntax of the pattern matching expression.

Understanding the New Keywords

The three new logical patterns act as building blocks for creating sophisticated logic. By combining these with existing patterns like relational, property, and type patterns, you can create highly readable conditional statements.

The and Pattern

The and keyword requires that both the preceding and succeeding patterns are satisfied. This is particularly useful for range checking. For example, validating that an integer is within a specific range used to require two separate comparisons. Now, it can be handled in a single line.

The or Pattern

The or keyword is satisfied if either the preceding or the succeeding pattern matches. This is a massive improvement over multiple equality checks. Instead of writing if (x == 1 || x == 2 || x == 3), you can now use a pattern matching approach that is cleaner and more declarative.

The not Pattern

The not keyword acts as a logical negation. It is often used to ensure that a value does not match a certain pattern, such as checking for non-null values or excluding specific types from a list. It simplifies scenarios where you want to perform an action on everything except a specific case.

Practical Use Cases and Examples

To truly appreciate the power of C# 9 logical patterns, let us look at real-world scenarios where these features shine. Consider a scenario where you are processing a temperature input. You need to ensure the value is within a safe operating range and not a sentinel value.

  • Range Validation: Using input is >= 0 and <= 100 ensures the value is valid.
  • Filtering Collections: When working with lists, you can filter objects using item is not null and { IsActive: true }.
  • Handling Multiple States: Using status is "Pending" or "Processing" simplifies state machine logic significantly.

The readability improvements are not just aesthetic. By reducing the cognitive load required to parse complex nested if-else structures, these patterns make code easier to maintain and debug. When a developer sees not null, the intent is immediately clear, reducing the chance of logic errors.

Comparing Old vs New Syntax

Consider a typical scenario where you are validating a user age. In older versions of C#, you might write: if (age >= 18 && age < 65) { /* logic */ }. While this is simple, as requirements grow—such as adding a check for a specific identification flag—the boolean logic becomes bloated. With C# 9, you can write: if (age is >= 18 and < 65). This declarative style reads almost like natural language, making the code self-documenting.

Furthermore, consider the not pattern. Previously, you might have written if (!(obj is string)). The new syntax allows for if (obj is not string), which is much more intuitive for developers coming from different backgrounds and improves the overall symmetry of the language.

Best Practices for Pattern Matching

While logical patterns are powerful, they should be used judiciously. Here are a few tips for maintaining clean code:

  • Keep patterns simple: Do not chain too many and/or operators together, as it can still become difficult to read.
  • Prioritize readability: If a pattern matching expression becomes longer than a single line, consider breaking the logic into a separate method or using a named variable.
  • Combine with Switch Expressions: Logical patterns are most effective when used within switch expressions, allowing for concise and exhaustive handling of different data scenarios.

Frequently Asked Questions

What versions of .NET support C# 9 logical patterns?

Logical patterns were introduced in C# 9, which is supported in .NET 5 and all subsequent versions, including .NET 6, 7, and 8. You must ensure your project is configured to use the C# 9.0 language version or higher.

Can I use logical patterns with property patterns?

Yes, absolutely. One of the greatest strengths of C# 9 is that you can combine property patterns with logical patterns. For example, you can check if an object's property matches a specific condition using person is { Age: > 18 and < 65 }.

Is there a performance penalty for using these new patterns?

No, there is no performance penalty. The C# compiler optimizes these logical patterns into the same underlying IL (Intermediate Language) code that would be produced by standard boolean operators. You get the benefit of cleaner, more maintainable code without sacrificing execution speed.

How does the not pattern handle nulls?

The not pattern is very effective for null checking. Using x is not null is a idiomatic and highly readable way to perform null checks in modern C#. It is semantically equivalent to checking x != null, but it integrates perfectly into larger pattern matching expressions.

Can I mix and and or in the same expression?

Yes, you can mix them. However, it is important to be mindful of operator precedence. The and keyword has a higher precedence than or. Always test your logic or use parentheses if you are concerned about the order of operations, just as you would with standard boolean logic.

Conclusion

C# 9 logical patterns represent a significant step forward in making C# a more expressive and modern language. By utilizing and, or, and not, developers can write code that is not only shorter but also easier to read and maintain. Whether you are validating data, filtering collections, or implementing complex business rules, these patterns provide the tools necessary to write cleaner, more robust applications. If you haven't started incorporating these patterns into your daily workflow, now is the perfect time to start.
